Release pipeline

① SYNC db.sh sync
→
② TEST Push staging
→
③ PREFLIGHT db.sh migrate
→
④ APPLY migrate --apply
→
⑤ DEPLOY merge → main
Step 1./scripts/db.sh sync — prod read only, staging replaced
Step 2Push staging → test /staging/
Step 3./scripts/db.sh migrate — no prod changes
Step 4./scripts/db.sh migrate --apply — quiet window required
Step 5Merge staging → main — smoke-test live

Command recipes

Deploy to staging

Weekly Safe
▼
URL/staging/
Writes prod?No

Option A — Automatic:

  1. Merge or push into staging branch
  2. GitHub Actions deploys automatically (~1–2 min)
  3. Smoke-test on /staging/

Option B — Manual: Actions → Deploy → Run workflow → target staging

Deploy to production

Release Frontend only
▼
  1. Confirm staging tests pass
  2. Merge staging → main
  3. Wait for deploy workflow
  4. Smoke-test: login, dashboard, one operational page

If release includes DB migrations, run migrate --apply during quiet window before users enter DSR data.

DB sync — prod → staging

Release Prod read-only
▼ 
./scripts/db.sh sync

Replaces all staging data. Requires scripts/db.env and Docker (or libpq).

DB migrate — preflight

Release No prod writes
▼ 
./scripts/db.sh migrate

Shows migration status and dry-run. Safe to run anytime.

DB migrate — apply (production)

Release Writes prod
▼ 
./scripts/db.sh migrate --apply

Auto-backups to scripts/.prod-backups/. Run during quiet window — no active DSR / day closing.

DB backup — local

Before risky ops Read-only
▼ 
./scripts/db.sh backup
